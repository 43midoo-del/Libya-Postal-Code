## 1. المرحلة 1 — قاعدة البيانات (`database.sql`)

### 1.1 الهدف من هذه المرحلة

إنشاء مخطط MySQL يلبّي متطلبات الـ SRS: مستخدمون بدور (مدير/موظف/مواطن)، تقسيم إداري (ولاية → شعبية → مدينة → منطقة)، وجدول عناوين يرتبط بالمنطكة والمُنشئ، مع قيود تمنع تكرار الكود البريدي أو **نفس زوج الإحداثيات** (منع تكرار الموقع على مستوى قاعدة البيانات).

### 1.2 ما تم إنشاؤه

- **اسم القاعدة:** `libya_postal`  
- **الترميز:** `utf8mb4` / `utf8mb4_unicode_ci` (دعم العربية وEmoji إن دعت الحاجة لاحقاً)  
- **محرك التخزين:** `InnoDB` (دعم المفاتيح الأجنبية transactions)

### 1.3 الجداول (شرح ميداني لكل جدول)


| الجدول      | الوظيفة                   | أهم الأعمدة                                                                                     |
| ----------- | ------------------------- | ----------------------------------------------------------------------------------------------- |
| `users`     | حسابات النظام             | `email` (فريد)، `password` (للتخزين بصيغة `password_hash`)، `role` كقيم `ENUM`                  |
| `states`    | المستوى الأعلى في التقسيم | `code` يُستعمل لاحقاً كـ **X** في `X-YY-ZZ-NNNN`                                                |
| `regions`   | تابع لولاية               | `state_id` مفتاح أجنبي                                                                          |
| `cities`    | تابع لشعبية               | `region_id`                                                                                     |
| `areas`     | تابع لمدينة               | `city_id` — أدنى مستوى نربط به العنوان مباشرة                                                   |
| `addresses` | عنوان فعلي على الخريطة    | `latitude` / `longitude` بصيغة `DECIMAL`، `postal_code` فريد، `area_id`، `created_by` → `users` |


### 1.4 القيود والفهارس (لماذا؟)

- **مفاتيح أجنبية** بين `states` → `regions` → `cities` → `areas` مع `ON DELETE RESTRICT` لمنع حذف تفكّ العناوين بصمت.  
- `**UNIQUE` على `addresses.postal_code`:** يضمن عدم توليد نفس السلسلة مرتين.  
- `**UNIQUE` على `(latitude, longitude)`:** يبرّر شرط “منع تكرار الموقع” تخزينياً (الدقة عشرية محدودة).  
- **فهارس** على `owner_name` و`postal_code` تسهّل البحث لاحقاً.

### 1.5 ما لم نُنفّذه هنا (عن قصد)

- إدخال **بذور (seed)** لحساب المدير: مؤجّل حتى اكتمال طبقة التحقق من كلمات المرور (المرحلة 3) حتى لا تُسجّل كلمات مرور “يدوية” بصيغة نص عادي.

### 1.6 الملفات


| المسار         | الدور                                   |
| -------------- | --------------------------------------- |
| `database.sql` | جمل `CREATE DATABASE` + الجداست الكاملة |

## 2. المرحلة 2 — الاتصال بقاعدة البيانات (PHP + PDO)

### 2.1 الهدف

توفير **نقطة وصول واحدة** لاتصال MySQL باستخدام PDO، مع عدم تكرار الاتصال (Singleton)، ووضع بيانات الاتصال في ملف `config` قابل للتعديل بيئةً بيئة (محلي / استضافة).

### 2.2 التنفيذ التقني

- **الصنف `App\Database`:** `includes/Database.php`  
  - `getInstance()`: يرجع نفس الـ instance.  
  - `getPdo()`: يعيد `PDO` ليستخدمه `Model` لاحقاً.
- **خيارات PDO المختارة**  
  - `ERRMODE_EXCEPTION`: أي خطأ SQL يصبح استثناءً (أسهل تتبعاً من القيم `false`.  
  - `FETCH_ASSOC` كافتراض: مصفوفات مرتبطة بالأعمدة.  
  - `EMULATE_PREPARES = false`: الاعتماد على تجهيز الاستعلامات الأصلي في MySQL عند التوافق.
- **ملف `config/database.php`:** يعيد مصفوفة (`host`, `port`, `database`, `username`, `password`, `charset`) — **يجب تعديله** ليتوافق مع XAMPP/WAMP/الخادم الحقيقي.  
- `**includes/bootstrap.php`:**  
  - ثابت `APP_ROOT`  
  - **تحميل تلقائي (autoload)** مبسّط لـ:  
    - `App\Models\...` → `models/...`  
    - `App\Controllers\...` → `controllers/...`  
    - `App\...` آخر (مثل `App\Database`, `App\Csrf`) → `includes/...`
  - **بدء الجلسة (session):** باسم مخصص من `config/app.php` بعد هذه المرحلة ليتم تمهيد **المرحلة 3**.
- `**index.php` (في بداية المشروع):** كان اختبار اتصال فقط. في **المرحلة 3** صار **موجّهًا أماميًا (front controller)** يرتبط بالجلسة والمتحكّمات (انظر أدناه).

### 2.3 المخاطر والأمان (ملاحظات تشغيل)

- **لا** تُرفع كلمات مرور قواعد بيانات حقيقية لأي مستودع عام.  
- في بيئة الإنتاج: صلاحيات MySQL **أدنى ما يكفي** لحساب التطبيق (لا `ALL PRIVILEGES` على `*` دون مبرر).

### 2.4 الملفات


| المسار                   | الدور                                                              |
| ------------------------ | ------------------------------------------------------------------ |
| `config/database.php`    | بيانات الاتصال                                                     |
| `config/app.php`         | إعدادات عامة (اسم التطبيق، اسم جلسة PHP) — أضيف/توسّع في المرحلة 3 |
| `includes/Database.php`  | Singleton + PDO                                                    |
| `includes/bootstrap.php` | autoload + session                                                 |
| `index.php`              | نقطة دخول التطبيق (تطوّر وصفها في §3)                              |

## 3. المرحلة 3 — تسجيل الدخول والجلسات والأدوار

### 3.1 الهدف

- تسجيل دخول ببريد إلكتروني وكلمة مرور.  
- تخزين كلمة المرور بصيغة **آمنة** (`password_hash` / `password_verify`) وليس نصاً صريحاً.  
- **جلسة PHP (session)** تحمل هوية المستخدم: المعرف، الاسم، البريد، **الدور** (`admin` / `employee` / `citizen`) ليُستند إليها لاحقاً في الصلاحيات.  
- واجهة **عربية RTL** بصفحة دخول وبداية “لوحة تحكم” بعد النجاح.  
- **CSRF** بسيط لنموذج تسجيل الدخول (حقل `csrf_token` + تحقق في السيرفر).

### 3.2 التصميم (MVC) في هذه المرحلة


| طبقة           | ماذا تفعل؟                                                          | ملفات                                                                                                     |
| -------------- | ------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| **Model**      | جلب المستخدم عبر البريد من قاعدة البيانات                           | `models/User.php` — `findByEmail`                                                                         |
| **Controller** | منطق النماذج: عرض تسجيل دخول، معالجة POST، تسجيل خروج، لوحة (عابرة) | `controllers/AuthController.php`, `controllers/DashboardController.php`, `controllers/BaseController.php` |
| **View**       | نماذج HTML/PHP: تسجيل دخول، لوحة، خطأ 403                           | `views/...` + `partials/head` و`foot`                                                                     |
| **دعم**        | جلسة، CSRF                                                          | `includes/SessionAuth.php`, `includes/Csrf.php`                                                           |


### 3.3 تدفق تسجيل الدخول (باختصار مُفصّل)

1. المستخدم يفتح `index.php?r=login` (أو `index.php` → يوجّهه النظام لنفس المسار حسب `home`).
2. يظهر `views/auth/login.php` مع `csrf_token` في الحقل المخفّي.
3. الإرسال: `POST` إلى `index.php?r=auth`.
4. `AuthController::processLogin` يصحح التوكن (CSRF)، يقرأ `email` و`password`، يستدعي `User::findByEmail`.
5. إن وُجد المستخدم: `password_verify` على `password` من النموذج مقابل `password` المخزّن (hash).
6. نجاح: `SessionAuth::login` يعيد **تشكيل** معرف الجلسة (`session_regenerate_id(true)`) ويخزن `id`, `name`, `email`, `role`.
7. إعادة توجيه لـ `index.php?r=dashboard`.
8. تسجيل الخروج: `?r=logout` يفريغ الجلسة ويُرجع لصفحة الدخول.

### 3.4 الأدوار (Role) في قاعدة البيانات والجلسة

- القيم في الجدول: `admin`, `employee`, `citizen` (نفس `ENUM` في `database.sql`).  
- **في هذه المرحلة** تُعرض الشارة النصية في اللوحة (مدير/موظف/مواطن)؛ **تقييد الصفحات حسب الدور** يتوسع عند إضافة شاشات (خريطة، عناوين) لاحقاً باستعمال `BaseController::requireAnyRole` أو فحص يدوي.

### 3.5 الوجه الأمامي (Front controller)

- المعلمة `?r=` في `index.php` توجه إلى:
  - `home`: إن كان مسجّلاً → لوحة، وإلا → دخول.  
  - `login`, `auth` (POST), `logout`, `dashboard`.
- أي `r` آخر: رسالة 404 بسيطة.  
- في حال استثناء غير متوقع: 500 ورسالة عربية؛ وإن وُضِع `APP_DEBUG=1` في البيئة (أو `?debug=1` اختياري) يظهر بيان تفصيلي للمطوّر فقط.

### 3.6 التنسيق

- `css/app.css`: ألوان داكنة، واجهة بسيطة، اتجاه RTL مضمون في `partials/head.php` بـ `dir="rtl"`.

### 3.7 بذور الاختبار (حساب المدير)

- الملف: `database_seed.sql` (يُنفَّذ **بعد** `database.sql`).  
- **للتطوير فقط** — **غيّر كلمة المرور في الإنتاج.**  
- بيانات الدخول الافتراضية (كما وردت في تعليق الملف):
  - البريد: `admin@libyapostal.local`  
  - كلمة المرور: `admin123`
- الـ hash مُخزّن مسبقاً ويتم التحقق منه عبر `password_verify` في PHP.

### 3.8 الملفات المضافة أو المعدّلة في هذه المرحلة


| المسار                                | الملاحظة                                              |
| ------------------------------------- | ----------------------------------------------------- |
| `index.php`                           | موجّه أمامي + سلوك `home`                             |
| `config/app.php`                      | اسم التطبيق، `session_name`                           |
| `includes/Csrf.php`                   | توليد/تحقق رمز                                        |
| `includes/SessionAuth.php`            | دالة تسجيل دخول/خروج، قراءة الدور                     |
| `models/User.php`                     | `findByEmail`                                         |
| `controllers/BaseController.php`      | `render`, `redirect`, `requireAuth`, `requireAnyRole` |
| `controllers/AuthController.php`      | تدفق الدخول                                           |
| `controllers/DashboardController.php` | لوحة بسيطة (تُبنى في المرحلة 4)                       |
| `views/auth/login.php`                | واجهة دخول عربية                                      |
| `views/dashboard/index.php`           | ترحيب + شارة الدور                                    |
| `views/partials/head.php`, `foot.php` | ترويسة/ذيل موحّدان                                    |
| `views/error/forbidden.php`           | قالب 403 (مستعد لاستعمال `requireAnyRole`)            |
| `css/app.css`                         | أنماط                                                 |
| `database_seed.sql`                   | مستخدم `admin`                                        |


### 3.9 كيفية الاختبار اليدوي

1. نفّذ `database.sql` ثم `database_seed.sql` في MySQL.
2. عدّل `config/database.php` لاستخدام قاعدتك.
3. شغّل مثلاً: `php -S localhost:8080` داخل مجلد المشروع.
4. افتح: `http://localhost:8080/index.php?r=login`
5. سجّل دخول بالبريد/كلمة أعلاه → يفترض أن تظهر اللوحة.
6. جرّب `?r=logout` ثم الرجوع لـ`dashboard` يجب أن يعيدك للدخول (حماية `requireAuth`).

## 4. المرحلة 4 — لوحة تحكم وإحصائيات مبدئية

### 4.1 الهدف

بعد دخول المستخدم: عرض **أرقام بسيطة** تعكس حجم البيانات الحالية (عدد العناوين، المستخدمين، الولايات المُعرّفة في الجدول الإداري) حتى تكون اللوحة “حية” حتى قبل إدخال عناوين فعلية (ستظهر أصفار عند بداية التشغيل). إضافة نصوص **روابط/ميزات قادمة** (خريطة، عناوين) لرسم المسار للمُقيّم أو فريق الاختبار.

### 4.2 التنفيذ

- **النموذج `App\Models\Statistics`:** `models/Statistics.php`  
  - دوال ثابتة: `countAddresses()`, `countUsers()`, `countStates()`  
  - تنفذ `SELECT COUNT(*)` فقط (قراءة بلا تعديل).
- `**DashboardController`:** يمرّر نتائج العدّ إلى `views/dashboard/index.php`.  
- **الواجهة:** شبكة `stat-grid` (ثلاث بطاقات) + تنسيق في `css/app.css` (`stat-card`، `stat-card__value`، …).  
- **إمكانية الخطأ:** الاستعلامات عبر جداول فاضية آمنة؛ العد = صفر. إن سقطت قاعدة البيانات، تُلتقط الاستثناءات في `index.php` (كما في المرحلة 2–3).

### 4.3 ماذا تُثبت هذه المرحلة؟

- أسلوب **MVC** واضح: “قراءة إحصاء” في `Model`، “تجميع” في `Controller`، “عرض” في `View`.  
- الاستعلام `COUNT(*)` خفيف ومناسب للوحة مبدئية (لا اختيار `*` كامل).

### 4.4 الملفات المضافة أو المعدّلة


| المسار                                | التعديل                                             |
| ------------------------------------- | --------------------------------------------------- |
| `models/Statistics.php`               | جديد                                                |
| `controllers/DashboardController.php` | تمرير `countUsers`, `countAddresses`, `countStates` |
| `views/dashboard/index.php`           | بطاقات إحصاء + “قريباً”                             |
| `css/app.css`                         | أنماط البطاقات والشبكة                              |

## 5. المرحلة 5 — دمج Leaflet (خريطة ليبيا + إحداثيات)

### 5.1 الهدف (حسب الـ SRS)

- عرض **خريطة تفاعلية** لِمنطقة ليبيا (ليس ميدان العالم كاملاً في التجربة).  
- **تقييد الحركة (pan)** داخل **مستطيل حدود تقريبي** (مستقيمات/دوائر) حول ليبيا؛ مع `maxBoundsViscosity` لمنع “الهروب” بعيداً عن المنطقة.  
- **النقر على الخريطة** يلتقط **خطي العرض/الطول** بنظام **WGS84** (نفس مخرجات GPS و`DECIMAL` في MySQL).  
- إظهار القيمة في حقول `input` جاهزة لربط **نموذج العنوان** (المرحلة 6).  
- **لا إطار عمل** على الواجهة: JavaScript عادي (Vanilla) كما فُرض في الـ stack.

### 5.2 اختيار التقنيات


| البند         | الاختيار                                      | السبب                                                         |
| ------------- | --------------------------------------------- | ------------------------------------------------------------- |
| مكتبة الخريطة | **Leaflet 1.9.4** (من CDN)                    | خفيفة، مستخدمة واسعاً، تتناسب مع “بدون build”                 |
| طبقات القاع   | **OpenStreetMap** (Raster)                    | بيانات مفتوحة + مناسب للمشروع الأكاديمي؛ مع إسناد رابط (حقوق) |
| إحداثي الحدود | مستطيل رقيق حول **ليبيا** في `config/map.php` | ليس “حدوداً سياسية دقيقة” بل **إطار عرض/تنقل** للتجربة        |


### 5.3 القيم الافتراضية (قابلة للتعديل)

الملف `config/map.php` يعرّف:

- `**libya_bounds`:** `south, west, north, east` (درجات عشرية WGS84).  
- `**default_center`:** مثال قرب **طرابلس** `[32.8872, 13.1913]`.  
- `**default_zoom` / `min_zoom` / `max_zoom`:** للتحكم في إظهار التفصيل (استكشاف/مناطق).

> إن وُضِع مركز افتتاحي **خارج** المستطيل (خطأ تكوين)، ينتقل `js/map.js` تلقائياً لمركز المستطيل أو `fitBounds` بعد الإنشاء.

### 5.4 طبقة الـ PHP (MVC)

- `**MapController`:** `requireAuth()` — الخريطة للمُستخدم **المسجّل** فقط (يمكن توسيع السياسة لاحقاً إن اقتضى الـ SRS نشراً أوسع).  
- يُحمّل `config/map.php` ويمرّر `bounds` والمركز والمستويات إلى `views/map/index.php`.  
- `**index.php`:** المسار `?r=map` يفتح الخريطة.  
- **لوحة التحكم:** رابط `فتح خريطة ليبيا (Leaflet)` يستدعي `index.php?r=map`.

### 5.5 طبقة الـ View

- `views/map/index.php`:  
  - يحقن **أوراق CSS** الخاصة بـ Leaflet عبر متغيّر اختياري `**$extraHead`** في `partials/head.php`.  
  - يضع `div#map-root` (مخفٍ) يحمل `data-*` فقط (لا يحتوي بيانات سرّية) لتمرير الإعدادات إلى JS.  
  - `div#map` للعرض الظاهر.  
  - حقول `#map-lat` و`#map-lng` (للقراءة فقط) تعرض 7 أرقام عشرية للتوافق المبدئي مع `DECIMAL(10,7)` في قاعدة البيانات.
- `**partials/head.php` و`foot.php`:** تدعم (`$extraHead`, `$extraFooter`) دون كسر باقي الصفحات.

### 5.6 طبقة الـ JavaScript

- **الملف `js/map.js`:**  
  - يبني `L.map` مع `maxBounds` و`maxBoundsViscosity: 1.0` ليُلزم التمرير **ضمن** الصندوق.  
  - يضيف `L.tileLayer` لـ OSM.  
  - على `map.on('click', …)`:  
    - يتحقق `bounds.contains(latlng)` (حماية إضافية).  
    - يضع/يحدّث `L.marker` عند النقطة.  
    - يملأ حقول `map-lat` / `map-lng` كسلاسل بـ `toFixed(7)`.

### 5.7 الأمان والاعتماد على الشبكة

- تحميل Leaflet + الأنماط من **CDN** (يتطلّب اتصالاً أثناء التطوير). للإنتاج دون إنترنت، يُنزَّع الـ `dist` داخل `public/` ويُضبط المسارات محلياً.  
- في **المرحلة 5** وحدها كانت إحداثيات **عرض** فقط؛ لاحقاً (المرحلة 6) تُرسل عبر `POST` مع **CSRF** وتحقق `GeoBounds` على السيرفر.

### 5.8 الاختبار اليدوي (قائمة تحقق)

1. `index.php?r=login` ثم الانتقال `index.php?r=map`.
2. التأكد أن **لا يُمكن** سحب الخريطة بعيداً جداً عن ليبيا (إحساس “أنت داخل صندوق”).
3. نقر مكانين مختلفين: يجب **تحريك** المؤشّر وتحديث الأرقام.
4. تغيير النافذة/الدقة: التأكد من أن `minZoom` / `maxZoom` يحترمان `config/map.php`.

## 6. المرحلة 6 — نموذج عنوان + حفظ + توليد بريد تلقائي

### 6.1 الهدف

- **إضافة عنوان** مرتبط **بمنطكة (`area_id`)** مع بيانات مالك ونوع و(اختياري) شقة.  
- **الإحداثيات** تُلتقط من **الخريطة** (Leaflet) كما في المرحلة 5.  
- **الحفظ** في MySQL بجدول `addresses` دون فضّ `postal_code` (NOT NULL) — لذلك أُدخلت طبقة **توليد فريد** بصيغة تقريبية `X-YY-ZZ-NNNN` مع **جدول عدّاد** `postal_counters`.  
- **الصلاحيات (SRS):** **الإدارة والموظف** يضيفان؛ **المواطن** يرى رسالة/403 عند المسار المباشر.

### 6.2 هجرة/بيانات إداريّة


| الملف                                   | محتوى                                                                                     |
| --------------------------------------- | ----------------------------------------------------------------------------------------- |
| `database_phase6_postal_and_sample.sql` | إنشاء `postal_counters` + بذور: ولاية/شعبة/مدينة/منطكة **واحدة (مثال)** + `INSERT IGNORE` |
| **تنفيذ يدوي**                          | بعد `database.sql` (وبعد المستخدم `database_seed` إن لزم) شغّل ملف الـ SQL أعلاه في MySQL |


**بدون** تلك البذور: قائمة `area_id` تكون **فارغة** — يتعطّل الحفظ (ويتم **تعطيل** زر الحفظ في الواجهة ورسالة واضحة).

### 6.3 صيغة الكود البريدي (توليد)

- **X:** الحرف الأول (ومجموعة ترمّز) من `states.code` (يدعم UTF-8 عبر `mbstring` إن وُجد، وإلا ASCII `substr`).  
- **YY:** `region_id % 100` مُبطّن بثنتين.  
- **ZZ:** `((city_id*3) + area_id) % 100` مُبطّن بثنتين (دمج “مدينة+منطقة” كما طُلِب مقاربةً).  
- **NNNN:** تسلسل **لكل `area_id`** عبر `postal_counters.last_n` داخل **معاملة** واحدة مع `INSERT` في `addresses` (— إن فشل الإدخال، `ROLLBACK` يعيد العدّاد).  
- **تفرد:** فهرس `UNIQUE(postal_code)` + منطق العدد؛ `UNIQUE(latitude,longitude)` يمنع تكرار **نفس الموقع** حتى عند تعديل مسمّى.

### 6.4 التحقق (سيرفر)

- **GeoBounds** (`includes/GeoBounds.php`): يردّ `true` فقط داخل `config/map.php[libya_bounds]` (يتسق مع الخريطة).  
- **بريد/نوع/مالك** في `App\Models\Address` مع رسائل عربية عند الخطأ.  
- **CSRF** على `address_store` مثل باقي النماذج.  
- **PRG:** إرسال POST ثم `redirect` إلى `?r=address_new` + رسالة `Flash` لمرة واحدة (`includes/Flash.php`).

### 6.5 المسارات

### 6.7 التذكير بما “دمج” مع المرحلة 7

- كانت خطة “المرحلة 7 = توليد بريد” **منفصلة** في الـ roadmap؛ **عمليّاً** أُدخلت **أساس التوليد** في المرحلة 6 لأن قاعدة البيانات تتطلّب `postal_code` **عند الحفظ**.  
- **المرحلة 7** لاحقاً: إن رغبتم: تحسين صيغة `YY/ZZ`، **استيراد** بيانات إدارية، أو واجهة لتحرير التسلسل.

### 6.8 الاختبار (يدوي)

1. تنفيذ `database_phase6_postal_and_sample.sql`.
2. دخول كمستخدم **موظف/مدير** → `?r=address_new`.
3. اختيار منطكة، **نقر** على الخريطة، تعبئة المالك، “حفظ”.
4. تظهر رسالة بنجاح مع **الكود البريدي** — واللوحة تُظهر (إحصاء) `COUNT(addresses)+1` بعد العودة.
5. إعادة حفظ **نفس النقطة** مرفوضة (قيد `UNIQUE` على الإحداثيات) برسلة عربية.

### 6.9 ما لحق بالمرحلة 7+ (لاحقاً)

## 7. المرحلة 7/8: البحث + الخريطة + تعديل/حذف

> نُنفّذ هنا **متطلبات الـ roadmap** (المرحلة 8: **بحث** بالكود/الاسم + **إظهار** على خريطة) مع **تعديل/حذف** (موظف/إدارة) لأنها مذكورة في “الميزات الأساسية” الأصلية ولا تحتمل تأخير واجهة الاستعمال.

### 7.1 من يصل لأي شيء؟

- **البحث وعرض النتائج والخريطة:** أي مستخدم **مسجّل الدخول** (يشمل `citizen` كما يسمح SRS بالبحث لصاحب العنوان/القارئ).  
- **تعديل/حذف:** فقط `admin` و`employee` (نفس فلسفة “إضافة عنوان”).

### 7.2 البحث

- **الواجهة:** `?r=search` — حقل `q` + `mode` (`all` / `code` / `name`) — إرسال **GET** (رابط قابل للمشاركة/الرجوع).  
- **النموذج:** `App\Models\AddressSearch`  
  - `LIKE` بسلسلة `%{q}%` بعد **إزالة** `%` و`_` و`\` من إدخال المستخدم (تبسيط أمني يقلل تعقيد `ESCAPE` في `LIKE`).  
  - حد `LIMIT 100` (ثابت `MAX_RESULTS`) كحماية من استعلامات مفرطة.
- **الخريطة:** `js/search_map.js` + Leaflet: يملأ `LayerGroup` بعلامات ويستدعي `fitBounds` على **نتائج البحث** فقط (لا يتطلب `maxBoundsViscosity` تفاعل نقر).  
- **تُحمّل** أوراق Leaflet/JS فقط عند **وجود نتائج** (لتجنب عناصر DOM ناقصة).

### 7.3 التعديل

- `?r=address_edit&id=` (GET) — نموذج `views/addresses/edit.php`  
- يُسمح بتعديل **المالك / النوع / البيانات الإضافية** فقط. **الكود البريدي والإحداثيات** ثابتان (تفادي إعادة توليد/تداخل) — **سياسة واضحة** لدراسة/عربون للتوسع.  
- `?r=address_update` (POST) + **CSRF** + `Flash` → إعادة توجيه لنفس `address_edit` مع رسالة.

### 7.4 ماذا عن “المرحلة 7” الأصلية في الـ roadmap؟

- **توليد البريد** وُلد فعليّاً في [§6](#6-المرحلة-6--نموذج-عنوان--حفظ--توليد-بريد-تلقائي) لأن `postal_code` إلزامي في `addresses`.  
- “المرحلة 7” **اختيارياً** أصبحت: **توسيع/تحسين** تسميات التقسيم، استيراد **GeoJSON**، **تحسين** واجهة إدخال/تحرير الترقيم.

### 7.5 الحذف

- `?r=address_delete` (POST) — جسم: `id` + `csrf_token` — **تأكيد** `confirm()` في العميل.  
- بعده `Flash` وإعادة توجيه لـ`search`.

### 7.6 الملفات


| المسار                                      | الدور                                                        |
| ------------------------------------------- | ------------------------------------------------------------ |
| `models/AddressSearch.php`                  | الاستعلام                                                    |
| `models/Address.php`                        | `findById` / `updateMeta` / `deleteById`                     |
| `controllers/SearchController.php`          | `search`                                                     |

### 7.7 اختبار يدوي

2. لاحق الخريطة تُركّز على النتائج.

4. جرّب حذف (تأكيد) ثم تأكد اختفاءه من `COUNT` في اللوحة.
