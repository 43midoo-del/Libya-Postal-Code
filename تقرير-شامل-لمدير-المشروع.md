# تقرير شامل لمدير المشروع — نظام إدارة العناوين البريدية الذكية (ليبيا)

**الغرض من هذا المستند:** عرض موجّه للإدارة يشرح بالتفصيل ما تم تنفيذه في المشروع: البنية، قاعدة البيانات، المسارات، المنطق البرمجي، الميزات، الأمان، وملفات الدعم (SQL، سكربتات، خرائط).  
**المجلد المعني بالتطبيق:** `Projict/` (جذر تشغيل الويب هو هذا المجلد مع `index.php`).  
**تاريخ إعداد التقرير:** مايو 2026.

---

## 1. ملخص تنفيذي

تم بناء **تطبيق ويب** بلغة **PHP 8+** على نمط **MVC يدوي** (بدون إطار عمل ثقيل)، مع قاعدة بيانات **MySQL** (`utf8mb4`)، وواجهة عربية **RTL**. النظام يوفّر:

- تسجيل دخول آمن بجلسات PHP وأدوار مستخدمين (`admin`، `employee`، `citizen`).
- لوحة تحكم تعرض إحصاءات قراءة فقط (مستخدمون، عناوين، ولايات في الجدول الإداري الهرمي).
- خريطة تفاعلية (**Leaflet**) لتثبيت موقع العنوان على **WGS84** ضمن حدود عرض تقريبية لليبيا.
- إضافة عناوين بريدية مع **توليد تلقائي للرمز البريدي** بصيغة خماسية المواءمة مع المحافظة/المنطقة/المدينة/القطاع/رقم العقار، مع عدّاد في قاعدة البيانات داخل **معاملة (transaction)**.
- بحث في العناوين (بالكود أو الاسم أو الاثنين) مع عرض النتائج على خريطة.
- تعديل بيانات وصفية للعنوان وحذفه (لأدوار الموظف والمدير فقط).
- حماية **CSRF** على النماذج الحساسة، وتحقق خادمي من أن الإحداثيات داخل صندوق ليبيا (`GeoBounds`).
- واجهة برمجية JSON داخلية (`address_api`) لعمليات إنشاء/تحديث/حذف/بحث/جلب سجل واحد لصفحة الإدارة المتقدمة.
- مصدر محلي لأسماء مدن داخل الشعبية (`shabiya_city_places` + `api_shabiya_cities`) لتفادي الاعتماد على شبكة خارجية في بيئة مثل XAMPP.

## 2. المتطلبات التقنية والتشغيل

| البند | القيمة |
|--------|--------|
| PHP | 8+ مع `pdo_mysql`، يُفضّل `mbstring` و`json` |
| MySQL | 8+ (أو 5.7+ مع `utf8mb4`) |
| الخادم | Apache (XAMPP) أو `php -S` للتطوير |
| الواجهة الخرائطية | Leaflet من CDN + طبقة OpenStreetMap |

**التشغيل السريع:** تعديل `config/database.php`، استيراد ملفات SQL (انظر القسم 6)، ثم فتح `index.php?r=login`. يوجد `run-server.bat` لتشغيل الخادم المدمج على المنفذ 8080.

**حساب تجريبي (بعد `database_seed.sql`):** البريد `admin@libyapostal.local` وكلمة المرور `admin123` — **يجب تغييرها في الإنتاج.**

## 4. المسارات (`?r=`) — جدول مرجعي كامل

يُمرَّر المسار عبر الاستعلام `index.php?r=...` كما في السطور التالية من `index.php`:

| القيمة `r` | الطريقة | المتحكم والدالة | من يصل؟ | الوظيفة |
|------------|---------|------------------|---------|---------|
| `home` | GET | مضمن في `index.php` | الجميع | إن كان مسجلاً → `dashboard`، وإلا → `login`. |
| `login` | GET | `AuthController::showLoginPage` | زائر | صفحة الدخول؛ إن كان مسجلاً يوجّه للوحة. |
| `auth` | POST | `AuthController::processLogin` | زائر | التحقق من CSRF ثم البريد وكلمة المرور (`password_verify`). |
| `logout` | GET | `AuthController::logout` | مسجّل | إنهاء الجلسة والعودة للدخول. |
| `dashboard` | GET | `DashboardController::index` | مسجّل | لوحة + أعداد. |
| `map` | GET | `MapController::index` | مسجّل | صفحة خريطة Leaflet لالتقاط إحداثيات. |
| `search` | GET | `SearchController::index` | مسجّل | بحث + جدول + خريطة إن وُجدت نتائج. |
| `address_new` | GET | `AddressController::newForm` | `admin`, `employee` | نموذج إضافة/إدارة عنوان (خريطة + حقول بريدية + ولاية/شعبية). يدعم `?id=` للتحرير المدمج. |
| `address_store` | POST | `AddressController::store` | موظف/مدير | حفظ عنوان عبر النموذج الكلاسيكي + Flash + PRG. |
| `address_edit` | GET | `AddressController::editForm` | موظف/مدير | يُحوّل إلى `address_new&id=` (مسار موحّد). |
| `address_update` | POST | `AddressController::update` | موظف/مدير | تحديث وصفي فقط ثم إعادة توجيه لصفحة الإدارة. |
| `address_delete` | POST | `AddressController::delete` | موظف/مدير | حذف السجل. |
| `api_shabiya_cities` | GET | `AddressController::apiShabiyaCities` | موظف/مدير (مسجّل) | JSON لمدن الشعبية من الجدول المحلي. |
| `address_api` | POST | `AddressController::api` | موظف/مدير | JSON: `action` = `create`, `update`, `delete`, `search`, `get`. |
| *(أي قيمة أخرى)* | — | مضمن في `index.php` | — | صفحة **404** عربية بسيطة. |

**معالجة الأخطاء:** أي استثناء غير ملتقط يعيد **500** مع رسالة عربية؛ إذا `debug` في `config/app.php` أو متغير البيئة `APP_DEBUG=1` أو `?debug=1` يُعرض نص الاستثناء للمساعدة في التشخيص.

## 5. قاعدة البيانات — الجداول والأعمدة المهمة

المخطط الأساسي في `database.sql` (قاعدة `libya_postal`):

### 5.1 `users`

- حسابات النظام: `email` فريد، `password` مخزّن كـ **hash** (`password_hash` في PHP)، `role` ∈ `admin | employee | citizen`.

### 5.2 التسلسل الإداري الهرمي (لربط العناوين بهيكل قابل للتوسعة)

| الجدول | العلاقة | ملاحظات |
|--------|---------|---------|
| `states` | الجذر | يحوي `code` استُخدم تاريخياً في صيغ بريدية؛ لا يزال الجدول موجوداً للإحصاء والتوسعة. |
| `regions` | تابع لـ `state_id` | |
| `cities` | تابع لـ `region_id` | |
| `areas` | تابع لـ `city_id` | أدنى مستوى في الهرم؛ عمود `area_id` في `addresses` يشير إليه. |

**ملاحظة تنفيذية حالية:** حقل **`area_id` عند الإنشاء** يُؤخذ من **`config/app.php` → `default_postal_area_id`** (قيمة افتراضية ثابتة مثل `1`) وليس من قائمة اختيار ديناميكية في الواجهة الحالية؛ هذا يبسّط النموذج ويربط الترميز الجغرافي بالشعبية والرمز الخماسي بينما يبقي ربطاً هرمياً في قاعدة البيانات للمستقبل.

### 5.3 `addresses`

يخزّن العنوان الفعلي:

- **موقع:** `latitude`, `longitude` — **قيود فريدة على الزوج** `(latitude, longitude)` لمنع تكرار نفس النقطة الجغرافية.
- **الرمز البريدي المعروض:** `postal_code` — فريد (`UNIQUE`).
- **تفكيك الرمز الخماسي (للتحليل والبحث):**  
  `pc_province` (B/T/F)، `pc_area`، `pc_city`، `pc_sector`، `pc_property`.
- **حقول وصفية إضافية:** `owner_name` (اختياري، nullable)، `type` (`residential`, `government`, `commercial`)، `apartment_number`.
- **ربط إداري جغرافي مع لوحة الإدخال:** `wilayah` (`barqa`/`tripolitania`/`fezzan`)، `shabiya` (اسم الشعبية بالعربية)، `locality`، `street_number`.
- **تدقيق:** `created_by` → `users.id`، `created_at`.

### 5.4 `postal_property_counters`

المفتاح المركّب: `(province_code, area_num, city_num, sector_code)` يخزّن **`last_property`** — آخر رقم عقار مُخصَّص.  
الزيادة تتم داخل المعاملة مع **`SELECT ... FOR UPDATE`** لتجنب ازدواجية تحت طلبات متزامنة.

### 5.5 `shabiya_city_places`

جدول **مدن وأماكن جاهزة** لكل شعبية (`shabiya_name`، اختياري `shabiya_code`، `place_name`، `lat`، `lng`، `place_kind`، `sort_order`)؛ يُعبأ عبر `database_seed_shabiya_cities.sql` أو توليد من `tools/emit_shabiya_cities_seed.php`.

## 7. المنطق البرمجي الأساسي (شرح الكود)

### 7.1 توليد الرمز البريدي — `PostalCodeService`

- **الصيغة النصية:**  
  `[Province] [Area]-[City]-[Sector] [Property]`  
  مثال: `B 2-1-S 9` حيث:
  - المحافظة: **B** برقة، **T** طرابلس، **F** فزان.
  - المنطقة والمدينة: أعداد صحيحة ضمن **1–999** (`validateAreaCity`).
  - القطاع: **حرف لاتيني واحد** A–Z (`normalizeSector`).
  - العقار: عدد صحيح يزيد تلقائياً حتى حد `MAX_PROPERTY` (999999).

- **`reserveNextProperty(PDO, ...)`:**  
  1. يحاول قراءة الصف مع **`FOR UPDATE`**.  
  2. إن لم يوجد صف، يُنشئ صفاً بـ `last_property = 1`.  
  3. إن وُجد، يزيد `last_property` بشرط عدم تجاوز الحد الأقصى.  
  4. يُرجع مصفوفة فيها `code` الكامل وجميع المكوّنات.

- **`wilayahKeyFromProvince`:** تربط B→`barqa`، T→`tripolitania`، F→`fezzan` لتتوافق مع حقول `addresses.wilayah` والتحقق من الشعبية.

### 7.2 إنشاء عنوان — `Address::create`

الترتيب:

1. تنظيف والتحقق من الطول والنوع؛ التحقق من تطابق **اسم الشعبية** مع **الولاية المشتقة من المحافظة** عبر `LibyaAdmin::isShabiyaInWilayah` (يمنع مثلاً اختيار شعبية شرقية مع محافظة فزان).
2. `GeoBounds::isInLibya` على الخادم (لا يعتمد على وحدة العميل فقط).
3. `beginTransaction`:
   - `PostalCodeService::reserveNextProperty` → يحدد الرمز والأجزاء.
   - `INSERT` في `addresses` مع تقريب الإحداثيات لسبعة منازل عشرية (`roundDecimal7`).
4. `commit`؛ عند فشل `PDO`: ترجمة بعض أخطاء التفرد إلى رسائل عربية (إحداثيات مكررة، تعارض بريدي).

### 7.3 تحديث وحذف — `Address::updateMeta` / `deleteById`

- **التعديل المسموح:** `owner_name`, `type`, `apartment_number` فقط — **لا يُعاد توليد الرمز ولا تُغيَّر الإحداثيات** من مسار التحديث الوصفّي (سياسة تثبيت للسلامة والامتثال).
- **الحذف:** حذف مباشر بالمعرف مع التحقق من أن الصف وُجد.

### 7.4 البحث — `AddressSearch::search`

- يزيل من مدخل المستخدم رموز `LIKE` الخاصة (`%`, `_`, `\`) لتقليل مخاطر سوء الاستخدام.
- أوضاع: `all` (كود أو اسم)، `code`، `name`.
- حد أقصى **`MAX_RESULTS = 100`**.
- يعيد أيضاً حقول `wilayah`, `shabiya`, `locality`, `street_number`, `pc_*` لعرضها في الجدول والخريطة.

### 7.5 التقسيم الإداري في الواجهة — `LibyaAdmin` + `config/libya_admin.php`

- يعرّف **3 ولايات تقليدية** و**22 شعبية** مع رمز مثل `B1`…`F22`.
- الدوال: التحقق من الولاية، تسمية العرض، التحقق أن الشعبية تابعة للولاية، واسترجاع صف الشعبية بالاسم العربي (لـ API المدن).

### 7.6 مدن الشعبية — `ShabiyaCity`

- استعلامات على `shabiya_city_places` بالاسم العربي للشعبية أو بالرمز `shabiya_code`.
- تُستخدم لتغذية قائمة اقتراحات أو تكبير الخريطة دون انتظار خدمات خارجية.

### 7.7 الإحصاءات — `Statistics`

- `COUNT(*)` على `users`, `addresses`, `states` — استعلامات خفيفة للوحة التحكم.

**ملف `config/postal_map_regions.php`:** يحدد مركزاً جغرافياً لكل منطقة مرقمة؛ عند النقر على الخريطة يمكن للواجهة اختيار **أقرب** مركز لتعبئة `pc_province` و`pc_area` (الرقم `n`) وربما الحقول الأخرى حسب منطق `address_create.js` — هذا يربط **النقر الجغرافي** بـ **مكوّنات الرمز**.

## 10. واجهة JSON الداخلية — `address_api`

- **POST** فقط، جسم JSON، يجب تضمين **`csrf_token`** صالحاً.
- **`action`:**
  - `create` — نفس حقول الإنشاء تقريباً كـ `store`؛ يعيد `record` كاملاً بعد الإنشاء.
  - `update` — تحديث وصفي بالمعرف.
  - `delete` — حذف بالمعرف.
  - `search` — `q` و`mode`.
  - `get` — جلب عنوان واحد بالمعرف (404 JSON إن لم يوجد).

الاستجابات: `{ ok, message?, ... }` مع ترميز Unicode غير مهرب للعربية.

## 12. الأمان والصلاحيات (ملخص للمراجعة الإدارية)

| الموضوع | التنفيذ |
|---------|---------|
| كلمات المرور | تخزين **مهاش فقط**؛ التحقق بـ `password_verify`. |
| الجلسات | اسم جلسة مخصص؛ تجديد معرف الجلسة بعد نجاح الدخول. |
| CSRF | على تسجيل الدخول، حفظ العنوان، التحديث، الحذف، وJSON API. |
| التفويض | `requireAuth()` للصفحات العامة؛ `requireAnyRole(['admin','employee'])` لإدارة العناوين؛ المواطن يصل للبحث والخريطة العامة وليس لإضافة عنوان. |

| SQL | استعلامات مُجهَّزة (`prepare`) مع معلمات ملزمة؛ تقليل حقن SQL. |
| البحث | تنقية مدخلات `LIKE`؛ حد أعلى للنتائج. |

**ثغرات مقصودة/محدودة في النطاق الأكاديمي:** لا يوجد في هذا الفرع نظام صلاحيات دقيقة لكل حقل، ولا سجل تدقيق (audit log)، ولا تشفير على مستوى التطبيق للبيانات الحساسة — يمكن تخطيطها لمراحل لاحقة.
