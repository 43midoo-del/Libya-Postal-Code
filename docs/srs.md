# 📄 وثيقة متطلبات النظام (SRS)

## 🏷️ اسم المشروع
نظام إدارة العناوين البريدية الذكي في ليبيا

---

# 1️⃣ المقدمة

## 1.1 الغرض من الوثيقة
تهدف هذه الوثيقة إلى تحديد وتحليل جميع متطلبات نظام إدارة العناوين البريدية، لتكون مرجعًا أساسيًا لفريق التطوير أثناء تنفيذ المشروع، وكذلك مرجعًا أكاديميًا لتقرير التخرج.

## 1.2 نطاق النظام
النظام عبارة عن تطبيق ويب يهدف إلى:
- إنشاء نظام عنونة بريدي موحد في ليبيا
- تحديد المواقع باستخدام الخريطة
- توليد كود بريدي تلقائي لكل موقع
- تسهيل البحث والوصول للعناوين

---

# 2️⃣ وصف عام للنظام

## 2.1 المشكلة
تعاني ليبيا من عدم وجود نظام عنونة بريدي دقيق، حيث يعتمد المستخدمون على وصف غير واضح مثل:
- "جنب الجامع"
- "قدام المحل"

وهذا يؤدي إلى:
- صعوبة تحديد المواقع
- تأخر خدمات التوصيل
- ضعف الاستجابة في الطوارئ

## 2.2 الحل المقترح
تطوير نظام يعتمد على الخريطة (GIS) لتحديد الموقع بدقة، وربطه بكود بريدي فريد.

## 2.3 المستخدمون
- مدير النظام (Admin)
- موظف (Employee)
- مواطن (Citizen)

---

# 3️⃣ المتطلبات الوظيفية (Functional Requirements)

## 3.1 نظام تسجيل الدخول
- تسجيل دخول باستخدام البريد وكلمة المرور
- تحديد صلاحيات المستخدم حسب الدور

## 3.2 إدارة المستخدمين
- إضافة مستخدم جديد (مدير)
- تعديل المستخدم
- حذف المستخدم

## 3.3 إدارة العناوين
- إضافة عنوان جديد
- تعديل عنوان
- حذف عنوان
- عرض تفاصيل العنوان

## 3.4 الخريطة
- عرض خريطة ليبيا فقط
- منع التفاعل خارج حدود ليبيا
- إمكانية التكبير التدريجي:
  - ولاية → شعبية → مدينة → منطقة
- تحديد الموقع بالنقر على الخريطة
- التقاط الإحداثيات (Latitude / Longitude)

## 3.5 توليد الكود البريدي
- توليد كود تلقائي عند اختيار الموقع
- الكود يكون فريد (غير مكرر)
- يعتمد على التقسيم الإداري

## 3.6 البحث
- البحث بالكود البريدي
- البحث باسم صاحب العنوان

## 3.7 لوحة التحكم
- عرض إحصائيات (عدد العناوين)
- توزيع العناوين حسب المناطق

## 3.8 إدارة الخريطة (للمدير)
- إضافة حدود الولايات
- إضافة حدود الشعبيات
- إضافة المدن والمناطق
- حفظها بصيغة GeoJSON

# 7️⃣ نظام الكود البريدي

## 7.1 الصيغة الأصلية (تاريخية للتوثيق فقط)
```
B-02-03-0001
```
هذه الصيغة وُضعت في أوّل صياغة للوثيقة، وتُحفظ هنا للرجوع التاريخي.

## 7.2 الصيغة المنفّذة فعلياً في النظام
الصيغة الخمسة-أجزاء المعتمدة في قاعدة البيانات وفي الواجهة:
```
X N-M-S P
```
مثال حقيقي:
```
B 2-1-S 9
```

### معاني الأجزاء
| الجزء | المثال | الوصف |
|------:|:------:|:------|
| `X`   | `B`    | رمز الولاية: **B** برقة، **T** طرابلس، **F** فزان |
| `N`   | `2`    | رقم الشعبية (1–22) المطابق لتقسيم 2007–2011 |
| `M`   | `1`    | رقم المدينة داخل الشعبية (1–999) |
| `S`   | `S`    | حرف لاتيني يمثّل الحي/القطاع داخل المدينة (A–Z) |
| `P`   | `9`    | الرقم التسلسلي للعقار داخل (X, N, M, S) — يُولَّد تلقائياً ويُقفل بقفل صفّ في جدول `postal_property_counters` لمنع التكرار |

### قواعد التفرّد
- العمود `postal_code` في جدول `addresses` فريد على مستوى قاعدة البيانات.
- زوج `(latitude, longitude)` فريد لمنع تسجيل عنوانين بنفس الإحداثيات.
- الرقم التسلسلي يُحجز داخل ترانزكشن (`SELECT ... FOR UPDATE`) لمنع تعارض المستخدمين المتزامنين.

### مثال تفصيلي
- الموقع داخل **شعبية بنغازي** (B5):
  - `B 5-1-S 9` = ولاية برقة، شعبية رقم 5، مدينة رقم 1، حي S، العقار رقم 9.
